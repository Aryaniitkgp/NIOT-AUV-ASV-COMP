The core structural difference from your drone case
Your drone stack works because you have three separable subsystems: perception (YOLO), a moving/known target (window, landing pad), and a controller mapping pixel error → actuator command. Underwater, the same triplet applies, but state estimation is the harder problem here, not perception. Drones usually have GPS as a backstop even indoors sometimes (or at least reliable visual odometry in open air). Underwater, you have none of that — no GPS, and even magnetometers are unreliable near the pool's metal structure (the source document actually calls this out: keep the IMU 18" from any metal). So the localization layer needs more care than in the aerial case.
1. State estimation (do this before any task-specific controller)

Depth (Z) — this one's actually easier than aerial altitude: pressure sensor gives absolute depth directly via p = ρgh, no drift, no fusion needed. Use it as ground truth for heave.
Orientation (roll/pitch/yaw) — IMU (accel+gyro+mag) through a Madgwick or Mahony filter, same as any aerial stack.
X/Y position — this is the real gap. No GPS analog. Two options:

Dead-reckon from commanded thrust + a drag model (cheap, drifts fast, fine for short task segments)
If you want it done properly: DVL (Doppler Velocity Log) if your sim/vehicle models one — several real SAVe teams in the source document used exactly this (SEDNA/SRM). Integrate DVL velocity for a much better position estimate than pure dead reckoning.


Landmark correction — treat each task target (buoy, gate, bin, octagon) as a known fixed landmark once detected, and use it to correct dead-reckoning drift when in view, same idea as fiducial-corrected indoor drone localization, just with your color blobs instead of ArUco tags.

Run all of this through an EKF: IMU as the prediction step, depth sensor + landmark detections as correction measurements. This is your one shared piece of infrastructure — every task below consumes this filtered state, it doesn't reinvent it.
2. Perception: what actually detects what

Path, buoys, bins, cupid panel — color segmentation is the right call here (not YOLO). The arena is deliberately built with bright, saturated, singular colors precisely so simple thresholding works — that's the actual design intent of the competition, and it's cheap enough to run at full frame rate with no training data. This is what I already built and tested.
When color breaks down — the "L"ove Lane gate is bare grey PVC pipe, no color to threshold on. That needs edge/line detection (Hough transform) or a small trained detector if color segmentation genuinely fails, not a bigger version of the same tool.
Obstacle avoidance — one correction to your framing: underwater, you don't get a depth camera the way you do in air. IR-based depth (RealSense-style) is basically useless underwater — water absorbs IR almost immediately. The real analog is stereo vision (if water clarity allows) or forward-looking sonar (more common in real AUVs, works regardless of turbidity). Worth flagging because "depth cam" doesn't port over as cleanly as it seems. That said — this specific ruleset doesn't actually require dynamic obstacle avoidance; the course geometry is fixed and known in advance, so "obstacle avoidance" here mostly reduces to staying inside known pool bounds, which is a much easier problem than general-purpose avoidance.

4. Mission sequencing
Everything above is a bag of independent controllers — a finite state machine (or behavior tree if you want more structure) is what actually runs the competition, transitioning on "task complete" or "timeout," matching the MVC-style Mission Planner architecture the source document itself describes (kernel / AUV Bridge / AUV EYE / Mission Task). This is the layer I hadn't built yet — it's the thing that would actually turn the perception module from before into a working run.
5. Validation order in sim specifically
Because you have zero sim-to-real transfer risk to worry about yet, the efficient order is: validate perception accuracy against Gazebo's ground-truth pose (you can literally diff your vision-estimated offset against the true model pose the simulator publishes — no other simulator gives you that for free), then tune each controller's gains in isolation against its own task, then chain into the FSM last. Don't tune the whole pipeline end-to-end first — same discipline as validating window-traversal before obstacle-avoidance before landing, individually, in your drone stack.
